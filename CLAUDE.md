# CLAUDE.md — wiki-quiz-kit

## Quick Start: `/ingest` Skill

The **`/ingest`** skill automates the entire pipeline. User shares a URL → one confirmation → everything else is automatic.

```
URL → Extract(multi-source+transcript) → Confirm → raw/ → wiki/ → permanent → questions → MOC → health
```

**Trigger:** User shares a video/article link, says "ingest", "摄入", "一键处理", or "compile this".

**The full pipeline only has 1 confirmation point** — after extraction, before writing files.

## Project Overview

This is a personal knowledge management system that treats knowledge work like a software build pipeline:

| Software Engineering | Knowledge Management | Implementation |
|---------------------|---------------------|----------------|
| `src/` | `raw/` | Unprocessed materials |
| `build/` | `wiki/` | Structured knowledge output |
| Compiler | LLM + prompts/ | raw → wiki transformation |
| IDE | Obsidian | Edit, view, navigate, link |
| Lint/CI | health-check.py | Link validity, consistency |
| Incremental compile | compile.py | Change detection, only process deltas |

## Architecture

```
raw/ (source) ──→ prompts/ (compiler passes) ──→ wiki/ (build output)
                        ↑
                    Claude Code (the compiler driver)

questions/ ──→ .claude/skills/review/ ──→ output/ (interactive quiz)
     ↑                                       │
     │                                  session JSON download
     │                                       │
     └── state/ ←────── sessions/ ←─────────┘
          (答题状态)      (答题记录WAL)

wiki/log.md (操作日志)  ←── /ingest, /review, /lint
wiki/purpose.md (方向)  ──→ read by all LLM operations
wiki/overview.md (总览) ←── /ingest, /review maintain
```

- **raw/:** Capture-first. Everything lands in `raw/inbox/`, gets triaged to subfolders.
- **wiki/:** Structured output. `permanent/` = atomic evergreen notes, `literature/` = reading notes, `daily/` = journal, `moc/` = Maps of Content.
- **prompts/:** LLM prompt templates for each compilation pass. Claude reads the source file, reads the prompt, then generates the output.
- **scripts/:** `health-check.py` audits vault integrity; `compile.py` detects changed files. All Python 3, cross-platform.

## Common Commands

```bash
# Python 3 — cross-platform (Windows, Mac, Linux)
python3 scripts/health-check.py              # summary only
python3 scripts/health-check.py --strict     # include symmetric link check
python3 scripts/health-check.py --verbose    # show per-issue details
python3 scripts/health-check.py --json       # JSON output for programmatic use

python3 scripts/compile.py                   # incremental scan
python3 scripts/compile.py --full            # treat all files as changed
python3 scripts/compile.py --what-if         # dry run — show what would happen
python3 scripts/compile.py --interactive     # step through each file
python3 scripts/compile.py --json            # JSON output (includes quizAffected)

python3 scripts/eval-llm.py --verbose        # validate generated questions
python3 scripts/eval-llm.py --output path    # specify output file to check

python3 scripts/validate-draft.py temp/draft-{slug}.json            # validate question draft
python3 scripts/validate-draft.py temp/draft-{slug}.json --write    # validate + write files

python3 scripts/clean.py --mode vtt --input file.vtt --output clean.txt   # clean VTT transcript
python3 scripts/clean.py --mode web --input page.md --output clean.md     # clean scraped web page
python3 scripts/clean.py --mode article --input article.md                # clean article text

python3 scripts/youtube-transcript.py <url>                    # fetch YouTube transcript to stdout
python3 scripts/youtube-transcript.py <url> -o file.txt        # fetch + write to file
```

## Note Types & Frontmatter Schema

Every note must have YAML frontmatter with a `type` field:

**Permanent note** (`wiki/permanent/`):
```yaml
type: permanent
tags: [...]
aliases: [...]      # alternate titles for linking
sources: []          # source note slugs that contributed
created: "YYYY-MM-DD"
modified: "YYYY-MM-DD"
```

**Literature note** (`wiki/literature/`):
```yaml
type: literature
sources: []          # source note slugs that contributed
source_url: ""       # original URL of the source material
author: ""
tags: [...]
created: "YYYY-MM-DD"
modified: "YYYY-MM-DD"
```

**Daily note** (`wiki/daily/`):
```yaml
type: daily
tags: [daily]
created: "YYYY-MM-DD"
```

**MOC** (`wiki/moc/`):
```yaml
type: moc
tags: [moc]
created: "YYYY-MM-DD"
modified: "YYYY-MM-DD"
```

**Fleeting note** (`raw/fleeting/`):
```yaml
type: fleeting
tags: []
created: "YYYY-MM-DD"
modified: "YYYY-MM-DD"
```

**Meeting note** (`raw/meetings/`):
```yaml
type: meeting
date: "YYYY-MM-DD"
attendees: []
tags: [meeting]
```

**Video note** (`raw/videos/`):
```yaml
type: video
sources: []             # source note slugs that contributed
source_url: ""         # original video URL
mirror_url: ""         # mirror URL (YouTube, B站, etc.)
transcript_url: ""     # URL used for transcript extraction (prefer transcript-capable platform)
transcript_platform: "" # platform that provided the transcript (YouTube, etc.)
author: ""
platform: ""           # 抖音, YouTube, B站, etc.
date: "YYYY-MM-DD"     # publish date
tags: []
created: "YYYY-MM-DD"
```

**Question** (`questions/`):
```yaml
type: question
id: "{topic}-{slug}-{n}"   # unique question identifier
topic: ""                   # primary topic tag for filtering
difficulty: ""              # easy / medium / hard
sources: []                # source note slugs that contributed
deprecated: false           # optional, marks retired questions (/review skips)
created: "YYYY-MM-DD"
```
> Question state (attempts, correct, wrong, last_result, last_attempt) is stored in `state/{id}.json`, NOT in the question frontmatter. Content is immutable after creation.

**Log** (`wiki/log.md`):
```yaml
type: log
created: "YYYY-MM-DD"
```

**Lint report** (`wiki/lint-{date}.md`):
```yaml
type: lint
date: "YYYY-MM-DD"
issues_found: 0
```

## Workflows Claude Should Execute

### 1. Inbox Triage
When the user says "triage my inbox" or points to `raw/inbox/`:
1. List all `.md` files in `raw/inbox/`
2. For each file, read the content
3. Apply the prompt from `prompts/inbox-triage.md`
4. Output the classification (DECISION, CONFIDENCE, REASON, SUGGESTED_TITLE, SUGGESTED_TAGS)
5. Ask user confirmation, then move the file to the suggested destination

### 2. Compilation
When the user says "compile this" or points to a raw/ file:
1. Read the source note from `raw/`
2. Read the appropriate prompt from `prompts/` based on the table below
3. Generate the output note under `wiki/`
4. Write to the correct wiki/ subfolder

**Source → Prompt → Output mapping:**

| Source | Prompt | Output |
|--------|--------|--------|
| `raw/inbox/*.md` | `prompts/inbox-triage.md` | Classify first, then compile |
| `raw/fleeting/*.md` | `prompts/fleeting-to-permanent.md` | `wiki/permanent/<slug>.md` |
| `raw/articles/*.md` | `prompts/article-to-literature.md` | `wiki/literature/<slug>.md` |
| `raw/highlights/*.md` | `prompts/highlight-to-permanent.md` | `wiki/permanent/<slug>.md` |
| `raw/meetings/*.md` | `prompts/meeting-to-notes.md` | polished version + permanent notes for insights |
| `raw/videos/*.md` | `prompts/video-to-literature.md` | `wiki/literature/<slug>.md` |

### 6. Quiz Review
When the user says "复习", "出题", "quiz", or "review":
1. Parse user intent into CLI args: topic tags, count, mode, difficulty
2. Run `python3 scripts/quiz-gen.py --tags <topic> --count <n> --mode <mode>` — zero LLM cost
   - `--mode random` (default): pure random from bank.json
   - `--mode new`: only unattempted questions
   - `--mode wrong`: only `last_result = "wrong"`
   - `--mode consolidate`: accuracy < 80%
3. Script reads bank.json, state/*.json, renders HTML, opens browser
4. After quiz, session JSON auto-downloads. State auto-updates via `watch-sessions.py`

### 3. Health Check (Tier 1 — deterministic)
When the user says "health check" or "check my vault":
1. Run `python3 scripts/health-check.py --verbose` — 8 checks including log integrity
2. Interpret the results in plain language
3. For each error, explain what's broken and suggest the fix
4. For orphans, suggest potential connection points
5. For state integrity errors, identify which concurrent writes failed
6. For source broken links, identify which wiki notes were deleted

### 3b. Semantic Lint (Tier 2 — LLM-driven)
When the user says "lint", "语义检查", "audit wiki":
1. Read `wiki/purpose.md`, `wiki/overview.md`, `wiki/log.md`
2. Sample wiki pages across topics
3. Check: contradictions, stale claims, missing concepts, orphan clusters, duplicates
4. Write `wiki/lint-{date}.md` report
5. Append to `wiki/log.md`
6. Present findings, ask which to fix

### 4. MOC Generation
When the user says "create an MOC for X":
1. Read `prompts/generate-moc.md`
2. Search vault for notes matching topic X (by title, tags, content)
3. Cluster found notes, identify entry points, identify gaps
4. Generate and write `wiki/moc/<topic-slug>.md`

### 5. Video Extraction (Multi-Source + Transcript Priority)

When the user shares a video link (Douyin, YouTube, B站, etc.), follow this priority chain:

**Phase 1: Multi-Source Discovery**
1. Extract the original video page content (tavily, web fetch)
2. Simultaneously, search for the video topic across platforms to find mirrors:
   - Search `"[video title]" + YouTube` to find YouTube mirrors
   - Search `"[video title]" + B站` for B站 mirrors
   - Search for companion articles (tech news, blog posts covering the same topic)
3. If a companion article/text report exists, extract it as `raw/articles/<slug>.md`

**Phase 2: Transcript Priority (CRITICAL)**
4. If a mirror is found on a transcript-capable platform (YouTube), fetch the page with transcript extraction
   - YouTube provides auto-generated transcripts via `tavily_extract` → use the transcript as the **primary raw material**
   - Transcript captures the creator's exact words with far more detail than video descriptions
5. If no transcript-capable mirror exists, fall back to the original platform's description

**Phase 3: Save & Cross-Reference**
6. Save the video note as `raw/videos/<slug>.md` with full metadata:
   - `source_url` = original video URL (even if platform blocks crawlers)
   - `mirror_url` = mirror URL on more accessible platforms
   - `transcript_url` = URL of the source used for transcript extraction
   - `transcript_platform` = platform name that provided the transcript
   - Include the full transcript text in the note body
   - **NO brief summaries** — the video note must preserve enough detail (core argument, supporting evidence, key data, reasoning structure) to be independently usable without re-watching the original video
7. If `raw/videos/` doesn't exist, create it
8. Offer to compile the video note into a literature note using `prompts/video-to-literature.md`

**Phase 4: Compilation**
9. When compiling, cross-reference all sources: transcript (primary), original description (secondary), companion articles (supplement)
10. Prefer direct quotes from the transcript over paraphrased claims

## Behavioral Rules

1. **Always use wikilinks** — `[[note-name]]` format, not `[text](path.md)`
2. **Always add frontmatter** — every note must have at least `type` and `created` fields
3. **Never delete raw/ source** — after compilation, the original raw/ file stays. The source is preserved for audit trail.
4. **One idea per permanent note** — if a fleeting note contains 3 distinct ideas, generate 3 permanent notes
5. **Flat permanent/ folder** — no subfolders in `wiki/permanent/`. Structure emerges from links, not folders.
6. **Titles as statements** — prefer "X causes Y under condition Z" over "About X and Y"
7. **When uncertain, stay in raw/** — if something might not warrant a permanent note, leave it in raw/ rather than polluting wiki/
8. **Video content fidelity** — never summarize a video with a brief 1-2 sentence blurb. Video notes must preserve the full reasoning chain (core argument, supporting evidence, key data points) so the knowledge is independently usable without re-watching the original.
9. **Git tracking default-off** — `raw/`, `wiki/`, `questions/`, `.obsidian/` are gitignored by default (they contain personal knowledge/quiz data). If the user asks to track these, warn them: pushing to GitHub will make their personal knowledge and answer history public. Recommend a private repository before un-ignoring.
